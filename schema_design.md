# 小说转剧本 YAML Schema 设计文档

## 版本：v1.0

---

## 一、Schema 完整定义

```yaml
script:
  # ── 元信息 ──
  meta:
    title: "小说名"                  # 剧本名称（取自原文或文件名）
    author: ""                       # 原作者（可留空）
    source_chapters: 3               # 处理的章节数（整数）
    total_scenes: 8                  # 拆分出的场景总数（整数）
    character_count: 5               # 提取的角色总数（整数）
    converted_at: "2026-06-05"       # 转换日期（YYYY-MM-DD）

  # ── 角色表（全局唯一 ID 索引） ──
  characters:
    - id: CHAR001                    # 全局唯一角色 ID，格式 CHAR + 三位数字
      name: "陈平安"                  # 角色姓名（必填）
      aliases: ["平安", "小平安"]     # 别名/昵称列表（可选，字符串数组）
      role: protagonist              # 角色类型（枚举）
      traits:                        # 性格/外貌特征标签（字符串数组）
        - "少年"
        - "坚韧"

  # ── 场景列表（按叙事顺序排列） ──
  scenes:
    - scene_id: SCENE_001            # 全局唯一场景 ID，格式 SCENE_ + 三位数字
      chapter: 1                     # 所属章节编号（整数）
      chapter_title: "第一章 笼中雀"  # 所属章节标题
      location: "泥瓶巷"             # 场景地点
      time_of_day: "清晨"            # 时间（枚举）
      characters_present:            # 本场景出场角色 ID 列表
        - CHAR001
        - CHAR002
      summary: "陈平安出门被邻居欺负"  # 场景一句话概要
      content:                       # 场景内容序列（有序列表）
        - type: action               # 内容类型（枚举）
          description: "陈平安推开门，阳光刺眼。"
        - type: dialogue             # 对话
          character: CHAR001         # 说话人 ID
          text: "我已经忍了三年。"    # 对话内容
          emotion: 平静              # 语气/情绪（可选）
        - type: narration            # 旁白/叙述
          description: "泥瓶巷的早晨一如既往地安静。"
```

---

## 二、字段枚举值

### role（角色类型）

| 值 | 含义 | 判断标准 |
|----|------|---------|
| `protagonist` | 主角 | 故事核心人物，贯穿全文 |
| `antagonist` | 反派 | 与主角对立的核心人物 |
| `supporting` | 配角 | 推动剧情的重要角色 |
| `minor` | 龙套 | 短暂出场的功能性角色 |

### type（内容类型）

| 值 | 含义 | 剧本用途 |
|----|------|---------|
| `action` | 动作/场景描写 | 指导演员走位、场景布置 |
| `dialogue` | 角色对话 | 演员台词，需标注说话人 |
| `narration` | 旁白/叙述 | 保留叙事氛围，可转画外音 |

### time_of_day（场景时间）

`清晨` | `上午` | `下午` | `傍晚` | `深夜` | `未知`

## 三、设计原则与理由

### 原则 1：角色与场景分离 —— 全局角色索引表

**设计**：`characters` 是顶层独立列表，场景通过 `characters_present: [CHAR001]` 引用角色 ID。

**理由**：
- **消除歧义**：同一角色可能在不同场景被不同称呼（"陈平安"/"平安"/"那少年"），通过 ID 引用确保唯一性
- **分段处理友好**：当小说过长需要分章节调用 LLM 时，先建立全局角色表，后续段落只需引用已有 ID，天然保持一致性
- **复用效率**：场记/选角导演只需看角色表就知道需要哪些演员，无需扫描全部场景
- **可扩展**：未来可以给角色添加更多字段（年龄、外貌、配音等），不影响场景结构

### 原则 2：三种内容类型（action / dialogue / narration）

**设计**：每个场景的 `content` 是一个有序列表，每个元素通过 `type` 字段区分类型。

**理由**：
- **精准覆盖剧本需求**：
  - `action` → 舞台指示、动作描写，告诉演员"做什么"
  - `dialogue` → 台词，标注说话人和情绪，演员可以直接读
  - `narration` → 叙述性文字，可转为画外音或删除
- **比"对话+动作"二分法更完整**：小说中存在大量纯叙述（环境描写、心理活动、转场过渡），这些既不是对话也不是动作，需要单独分类
- **下游灵活性**：下游工具可以按需过滤——配音演员只看 dialogue，导演看 action 走位，编剧看 narration 判断是否保留

### 原则 3：场景粒度 = 地点或时间的显著变化

**设计**：每当地点发生显著变化，或时间发生明显跳跃，即触发新场景。

**理由**：
- **符合影视工业标准**：影视剧本的分场惯例就是按"同一时间、同一地点"划分
- **可操作性**：每个场景对应一次实际拍摄设置（灯光、道具、演员调度），粒度适中
- **避免过细或过粗**：如果按段落分场景会过碎，如果按章节分又太粗——"地点+时间"是最自然的分割点

### 原则 4：characters_present 前置

**设计**：每个场景开头列出 `characters_present`，再展开 `content`。

**理由**：
- **场记效率**：快速确认"这场需要叫谁来"，不需要扫描 content 列表
- **完整性校验**：validator 可以检查出场角色是否都在角色表中注册，反之也能发现角色表中"从未出场"的冗余角色
- **拍摄计划生成**：未来可以根据此字段自动生成"按角色聚合的拍摄日程表"

### 原则 5：role 使用枚举值而非自由文本

**设计**：`role` 只能是 `protagonist` / `antagonist` / `supporting` / `minor` 之一。

**理由**：
- **下游可编程**：工具可以自动统计主角出场率、配角分布等
- **避免歧义**：如果允许自由文本，同一个角色可能被标为"主要人物"/"主角"/"核心"，无法做结构化查询
- **够用不冗余**：四个枚举值覆盖了 99% 的小说角色分类需求

## 四、Schema 扩展性说明

当前 Schema 为 v1.0，设计时预留了扩展空间：

| 扩展方向 | 实现方式 | 示例 |
|---------|---------|------|
| 新增角色字段 | characters 列表元素加键值对 | `age: 25`, `gender: 男`, `voice: 低沉` |
| 新增内容类型 | type 枚举扩展 | `type: monologue`（独白）、`type: flashback`（闪回） |
| 新增场景字段 | scenes 列表元素加键值对 | `weather: 雨`, `mood: 紧张`, `camera: 近景` |
| 多作品关联 | meta 加嵌套对象 | `series: { name: 三部曲, volume: 1 }` |
| 导出其他格式 | 基于 Schema 做转换器 | YAML → Final Draft / Fountain / 纯文本剧本 |

所有扩展均向后兼容——现有字段不变，新字段为可选，旧版本解析器不会因未知字段而报错。

## 五、与同类方案对比

| 维度 | 本 Schema | 纯文本剧本 | 对话-动作 二元格式 |
|------|----------|-----------|-------------------|
| 角色复用 | ✅ 全局 ID 索引 | ❌ 每次写名字 | ❌ 名字可能不一致 |
| 结构化查询 | ✅ YAML 可编程解析 | ❌ 需 NLP 再处理 | ⚠️ 部分可解析 |
| 影视制作适配 | ✅ 场景/角色/类型分离 | ❌ 需人工整理 | ⚠️ 缺少旁白分类 |
| 长文本处理 | ✅ 分段合并 ID 一致 | N/A | ❌ ID 可能冲突 |
| 人机可读 | ✅ 文本编辑器 + 代码工具 | ✅ 仅人可读 | ⚠️ JSON 对人不够友好 |
